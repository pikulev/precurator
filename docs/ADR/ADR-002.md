# ADR 002: Спецификация API библиотеки `precurator` (Cybernetic Control for AI Agents)

**Статус:** Final / Approved with refinements  
**Автор:** System Architect / Control Theory Expert  
**Технологический стек:** TypeScript, LangGraph, Zod, OpenTelemetry.

---

## 1. Концепция (Philosophy)
Библиотека должна предоставлять **декларативный интерфейс** для описания систем управления и **императивные точки расширения** для реализации конкретной логики. Вместо "черного ящика", API предоставляет каркас (Scaffold), который навязывает разработчику дисциплину "Прогноз-Корректор".

Ключевой принцип после ревью: описание системы должно существовать как **чистый сериализуемый конфигурационный объект**, а builder-API может выступать только как удобная оболочка над ним.

`ControlSystemConfig` является **canonical source of truth**. Любая возможность, доступная через `ControlSystemBuilder`, обязана быть выразима через сериализуемый config; builder не должен вводить скрытую императивную семантику или дополнительные возможности, отсутствующие в декларативной модели.

После ревью важно зафиксировать границу: сериализуемый `ControlSystemConfig` описывает **что** должно быть собрано, а runtime registries/adapters описывают **чем** именно это будет исполнено в конкретном процессе. Инстансы моделей, функций, клиентов SDK и конкретных tool handlers не должны считаться частью канонического JSON-ready конфига; они подключаются по ссылкам, идентификаторам или через внешний реестр исполнения.

## 2. Требования к State (Пространство состояний)

API должно гарантировать, что состояние системы (Graph State) является расширяемым, но содержит обязательный "Кибернетический базис". После ревью state должен быть разделен на:

*   `control` — доменные данные и вычислимые сигналы регулятора.
*   `runtime` — процессные метаданные исполнения, HITL, checkpoints и telemetry.

### 2.1. Контракт состояния
Библиотека предоставляет базовый тип `ControlState<TTarget, TCurrent>`, который разработчик расширяет своими данными.
```typescript
type StructuredErrorVector = Record<string, number | string | boolean | null>;

type ErrorTrend = 'improving' | 'flat' | 'degrading' | 'oscillating';
type CompactionStrategy = 'sliding-window' | 'summarize-oldest' | 'hybrid';
type SummaryReplacementSemantics = 'replace-compacted-steps' | 'replace-all-but-recent-window';

interface ShortTermMemoryWindow {
  steps: ControlStep[];
  maxShortTermSteps: number;
  compactionStrategy: CompactionStrategy;
  summaryReplacementSemantics: SummaryReplacementSemantics;
  summary?: string;
}

interface CompactionContext {
  maxShortTermSteps: number;
  strategy: CompactionStrategy;
  semantics: SummaryReplacementSemantics;
  auditLogRef?: string;
}

type CompactShortTermMemory = (
  history: ControlStep[],
  context: CompactionContext
) => ShortTermMemoryWindow;

interface ControlBasis<TTarget, TCurrent> {
  target: TTarget;                        // Вектор целей (Set Point)
  current: TCurrent;                      // Текущий вектор состояния
  worldContext: Record<string, unknown>;  // Модель мира / знания / ограничения
  errorVector: StructuredErrorVector;     // Машиночитаемая ошибка
  errorScore: number;                     // Нормированная ошибка [0, 1]
  deltaError: number;                     // Изменение ошибки относительно предыдущего k
  errorTrend: ErrorTrend;                 // Тренд сходимости / деградации
  shortTermMemory: ShortTermMemoryWindow; // Bounded/reducible память регулятора
  prediction?: string;                    // Ожидаемый результат (Mental Model)
}

interface RuntimeContext {
  k: number;                              // Дискретное время (номер итерации)
  status: 'optimizing' | 'converged' | 'stuck' | 'failed' | 'awaiting_human_intervention' | 'aborted';
  stopReason?: string;                    // Причина завершения цикла или паузы
  diagnostics?: {
    code: string;
    message: string;
    evidence?: Record<string, unknown>;
    lastRepeatedAction?: string;
    oscillationWindow?: number[];
    recommendedAction?: 'retry-conservative' | 'interrupt' | 'abort' | 'human-override';
  };
  auditLogRef?: string;                   // Внешний журнал, не попадающий в prompt
  checkpointId?: string;                  // Текущий checkpoint
  bestCheckpointId?: string;              // Точка отката к лучшему состоянию
  tokenBudgetUsed?: number;               // Накопленный budget, израсходованный guard-логикой
  simulation: boolean;                    // Признак ветки симуляции
  metadata?: Record<string, unknown>;     // trace_id, user_id, thread_id и др.
  humanDecision?: Record<string, unknown>;
}

interface ControlState<TTarget, TCurrent> {
  control: ControlBasis<TTarget, TCurrent>;
  runtime: RuntimeContext;
}
```

Полный лог не должен автоматически попадать в контекст LLM. История для reasoning и аудит должны быть разведены на уровне контракта.

Правило после ревью: `runtime`-поля не являются источником истины о состоянии предметной области. Бизнес-логика не должна опираться на telemetry IDs, checkpoint IDs или trace metadata там, где требуется сравнение `target` и `current`.

Контракт compact/summarization должен быть явным extension point. Если библиотека поддерживает сворачивание памяти, она обязана либо принимать `compactShortTermMemory`, либо предоставлять эквивалентную декларативную стратегию с теми же гарантиями: детерминированность, отсутствие side effects, явная политика замещения и сохранение связи с внешним audit trail.

## 3. Архитектура API: Core Controller

Центральным элементом является декларативный объект `ControlSystemConfig`, который компилируется в `StateGraph`. `ControlSystemBuilder` допускается как ergonomic API, но не как единственная форма описания системы.

### 3.1. Инициализация и Типизация
```typescript
const config: ControlSystemConfig<MyTarget, MyCurrent> = {
  schemas: {
    target: z.object({ ... }),
    current: z.object({ ... })
  },
  modelRef: 'gpt-primary',
  stopPolicy: {
    epsilon: 0.05,
    maxIterations: 10,
    maxTokenBudget: 50_000
  },
  mode: 'conservative'
};

const system = compileControlSystem(config, {
  models: {
    'gpt-primary': llmInstance
  }
});

// Допустимая ergonomic-оболочка над тем же config
const sameSystem = ControlSystemBuilder.fromConfig(config).compile({
  models: {
    'gpt-primary': llmInstance
  }
});
```

`compileControlSystem(config)` должен рассматриваться как канонический путь. Builder-first стиль допустим только как синтаксический сахар над тем же сериализуемым описанием.

Нормативное правило после ревью: сериализуемый config хранит ссылки (`modelRef`, `toolRefs`, `verifierRef`, `promptVersion`, `policyRefs`), а `compileControlSystem(..., runtimeRegistry)` разрешает их в конкретные инстансы и функции текущего процесса.

Publish-ready требования к контракту конфигурации:

1.  **Полная типизация generics:** `TTarget` и `TCurrent` должны без потери типов проходить через `Observer`, `Comparator`, `Predictor`, `Verifier` и пользовательские расширения состояния. Использование `any` в публичных API-контрактах считается нарушением спецификации.
2.  **Runtime validation через схемы:** `schemas.target` и `schemas.current` не являются декоративными. Библиотека обязана валидировать входные значения и обновления `current` до начала reasoning-цикла; при нарушении схемы должна возвращаться понятная ошибка уровня контракта, а не поздний сбой внутри LLM/tool execution.
3.  **JSON-ready config:** Канонический `ControlSystemConfig` должен быть представим как чистый JSON-документ, за исключением ссылок-идентификаторов, которые затем разрешаются runtime registry.
4.  **Явные extension points:** Если библиотека допускает imperative shortcuts, они обязаны сводиться к тому же декларативному контракту и не могут вводить скрытое состояние, недоступное через сериализуемый config.

### 3.2. Регистрация узлов (ПФУ Components)
Каждый узел должен иметь четко определенный контракт:

*   **Observer:** `({ executionContext }) => Promise<TCurrent>` — получение данных из среды.
*   **Comparator:** `(target, current, context) => Promise<{ errorVector, errorScore, deltaError, errorTrend }>` — расчет рассогласования и его динамики.
*   **Predictor (Strategist):** Выбор действия на основе краткосрочной памяти, прогноза и world model.
*   **Effector:** Регистрация инструментов (Tools).
*   **Verifier:** Независимая проверка фактического прогресса, анти-осцилляционной логики и условий для `interrupt`.

Каждый tool handler должен принимать `executionContext` как минимум с полями `simulation`, `readOnly`, `traceId` и `checkpointId`. Разрушительные инструменты обязаны либо поддерживать `dryRun`, либо быть несовместимыми с simulation branch на уровне регистрации/типов.

Для publish-ready версии дополнительно требуется, чтобы все node contracts оставались checkpointer-safe: обработчики могут быть зарегистрированы во внешнем runtime registry, но их результаты, входы и возвращаемые структуры, попадающие в state, должны оставаться сериализуемыми и пригодными для возобновления через LangGraph checkpointer без ручной "реанимации" окружения.

```typescript
const system = compileControlSystem({
  ...config,
  observerRef: 'cloud-observer',
  comparator: ({ target, current, previousErrorScore }) => {
    return deterministicDiff(target, current, previousErrorScore);
  }, // Базовый путь: детерминированная функция, а не обязательно LLM
  verifierRef: 'external-verifier',
  toolRefs: ['tool1', 'tool2']
}, {
  observers: {
    'cloud-observer': async ({ executionContext }) => {
      return fetchObservedState(executionContext);
    }
  },
  verifiers: {
    'external-verifier': externalVerifier
  },
  tools: {
    tool1,
    tool2
  },
  tokenBudgetEstimator: ({ current, comparison }) => {
    return estimateStructuredTokenCost(current, comparison);
  }
});
```

## 4. Расширенные возможности (Control Theory Features)

### 4.1. Simulation Mode (Прогноз перед действием)
API должно поддерживать "воображаемые итерации".
```typescript
system.enableSimulation({
  depth: 1, // На сколько шагов вперед LLM должна просимулировать результат
  model: simulationLLM, // Можно использовать модель подешевле
  isolatedState: true,  // Симуляция не пишет в основной runtime history
  sideEffects: 'forbid', // Реальные инструменты запрещены, только dry-run/mock
  traceNamespace: 'simulation'
});
```

Симуляция обязана работать на ветвящейся копии состояния. Любые записи в журнал, checkpoints и telemetry должны явно помечаться как `simulation`.

Если инструмент не поддерживает dry-run/mock-режим, он не должен быть доступен в simulation branch. Признак симуляции хранится в `runtime.simulation`; если библиотека дополнительно использует execution context или event envelope, это должно быть согласовано с тем же флагом и не должно расходиться с runtime state.

Для устранения двусмысленности между graph state и вызовами инструментов библиотека должна пробрасывать этот признак и в `executionContext.simulation`. Несогласованность между `runtime.simulation` и execution context считается ошибкой инфраструктуры выполнения.

### 4.2. Stability Guards (Защита от дребезга)
Механизмы, предотвращающие бесконечные осцилляции.
```typescript
system.addGuard('anti-oscillation', {
  threshold: 3, // Если ошибка растет 3 раза подряд - сменить стратегию
  metric: 'deltaError',
  verifier: externalVerifier,
  onViolation: (state) => { /* Аварийный сценарий: conservative mode или interrupt */ }
});
```

Guard не должен опираться только на самооценку LLM-предиктора; нужен внешний наблюдатель, правило или формальная метрика. Решения guards должны учитывать не только абсолютный `errorScore`, но и `deltaError`/`errorTrend`.

### 4.3. Human-in-the-loop / Interrupts
Для high-risk систем библиотека должна поддерживать внешнее вмешательство человека как часть штатного control flow.

```typescript
system.enableHumanIntervention({
  onInterrupt: async (snapshot) => await notifyOperator(snapshot),
  allowedDecisions: ['resume', 'abort', 'override'],
  requiredWhen: ['policy_violation', 'critical_drift', 'destructive_action']
});
```

Минимальный контракт после ревью:

*   `interrupt` — ставит граф на паузу и переводит `runtime.status` в `awaiting_human_intervention`.
*   `resume` — продолжает цикл с новым `k` и сохранением audit trail.
*   `abort` — завершает цикл с `runtime.status = 'aborted'`.
*   `humanDecision` — внешнее решение оператора, сохраняемое в `runtime`.

При интеграции с LangGraph реализация паузы/возобновления должна опираться на штатный `checkpointer`. Библиотека не должна вводить альтернативный источник истины для suspended state вне checkpoint-механизма; она лишь оборачивает его ergonomic API для `approve`, `override`, `resume` и `abort`.

## 5. Observability и Телеметрия

Библиотека должна быть "прозрачной" для мониторинга.

### 5.1. Интеграция с LangSmith
Все внутренние переходы и вызовы LLM должны иметь автоматические теги:
*   `control_step_type`: (Observation | Comparison | Prediction | Execution)
*   `error_score`: текущее значение
*   `delta_error`: изменение ошибки относительно предыдущего шага
*   `error_trend`: направление динамики ошибки
*   `diagnostics_code`: код диагностического события при stuck/failure/interrupt
*   `simulation`: признак simulation branch
*   `checkpoint_id`: активный checkpoint для трассировки pause/resume
*   `control_component`: конкретный внутренний компонент (`observer`, `comparator`, `predictor`, `verifier`, `effector`)

### 5.2. Event Bus
Разработчик может подписаться на события системы для построения UI/графиков:
```typescript
system.on('step:completed', (event) => {
  console.log(
    `Iteration ${event.k}: error=${event.errorScore}, delta=${event.deltaError}, trend=${event.errorTrend}`
  );
});
```

Дополнительно система должна публиковать событие `step:interrupted`, если `Verifier`, guard или `StopPolicy` эскалируют управление к человеку.

Минимальный набор telemetry-полей после ревью: `thread_id`, `checkpoint_id`, `best_checkpoint_id`, `stop_reason`, `simulation`, `token_budget_used`, `delta_error`, `error_trend`, `human_intervention_reason`.

## 6. Дистрибуция и Developer Experience

Publish-ready пакет должен быть простым в установке, прозрачно интегрироваться в проект пользователя и не тащить неявную инфраструктурную сложность.

1.  **Peer dependencies:** `langgraph` и `@langchain/core` должны декларироваться как `peerDependencies`, чтобы библиотека не форсировала конфликтующие копии ключевых runtime-пакетов в приложении пользователя.
2.  **Минимальный install footprint:** Пакет не должен подтягивать тяжелые UI-kit, провайдеры LLM или иные необязательные рантаймы без явного запроса разработчика.
3.  **One-minute guide:** Публичная документация обязана содержать минимальный сценарий "Hello World", который демонстрирует `compileControlSystem(config)` без сложного доменного окружения.
4.  **API reference:** Все поля `ControlBasis`, `RuntimeContext`, `ControlSystemConfig` и runtime registry contracts должны быть документированы в reference-формате, пригодном для проверки инвариантов, а не только для маркетингового обзора.
5.  **Пресеты:** Библиотека должна поставлять хотя бы один `DefaultComparator` для детерминированного числового/структурного diff и один `DefaultSummarizer` для bounded memory, чтобы путь "из коробки" не требовал от пользователя сразу писать всю механику вручную.
6.  **Dual package support:** Дистрибутив должен поддерживать современный ESM и совместимый CommonJS-выход, если библиотека позиционируется как reusable npm-package для смешанных Node.js-окружений.

## 7. Жизнеспособность: Пример реализации сложной задачи
**Задача:** Автономное приведение облачной инфраструктуры (Terraform) к заданному вектору безопасности.

```typescript
// 1. Определение целей (Target Vector)
const target = {
  openPorts: [80, 443],
  unusedResources: 0,
  complianceLevel: "HIPAA"
};

// 2. Декларативная конфигурация системы
const infraConfig: ControlSystemConfig<InfraTarget, InfraState> = {
  schemas: { target: TargetSchema, current: CurrentStateSchema },
  observerRef: 'cloud-observer',
  comparator: deterministicInfraComparator,
  verifierRef: 'external-security-verifier',
  toolRefs: [
    'close-port',
    'terminate-instance',
    'update-policy'
  ],
  predictor: {
    reasoning: "chain-of-thought",
    predictionMandatory: true
  },
  stopPolicy: {
    epsilon: 0.05,
    maxIterations: 10,
    maxTokenBudget: 50_000
  },
  humanIntervention: {
    requiredWhen: ['destructive_action', 'policy_violation']
  },
  simulation: {
    isolatedState: true,
    sideEffects: 'forbid'
  }
};

const infraManager = compileControlSystem(infraConfig, {
  observers: {
    'cloud-observer': async ({ executionContext }) => await cloudProvider.fetchCurrentState(executionContext)
  },
  verifiers: {
    'external-security-verifier': externalSecurityVerifier
  },
  tools: {
    'close-port': new ClosePortTool(),
    'terminate-instance': new TerminateInstanceTool(),
    'update-policy': new UpdatePolicyTool()
  }
});

// 3. Запуск итеративного процесса
const finalReport = await infraManager.invoke({
  target,
  metadata: { thread_id: "run-42", user_id: "secops" }
});
```

## 8. Эволюция API (Non-breaking changes)

1.  **Версионирование промптов:** Библиотека хранит версии системных промптов для узлов `Comparator` и `Predictor`. Разработчик может переопределять их, не ломая логику графа.
2.  **Middleware:** В будущих версиях добавится `withMiddleware((state) => ...)` для инъекции логики (логирование, модификация вектора ошибки) между любыми узлами.
3.  **Checkpoints / Time Travel:** API будет развиваться в сторону нативного доступа к checkpoint history и откату к минимальной ошибке.

## 9. Итоги ревью (Design Rationale)

*   **Type Safety:** Использование Generics `<TTarget, TCurrent>` исключает передачу несовместимых данных между узлами, а runtime validation через Zod не позволяет некорректному `current` дойти до LLM-цикла.
*   **Decoupling:** Логика описания системы отделена от механики ее сборки, доменное состояние отделено от runtime-метаданных, а история reasoning отделена от внешнего audit trail и runtime registry references.
*   **Resilience:** `StopPolicy`, checkpoints, HITL interrupts, внешний verifier, bounded memory, simulation isolation и структурированный `errorVector` предотвращают "зависание" агентов и позволяют анализировать сходимость без парсинга текста.
*   **Operational Fit:** Peer dependencies, dual-package distribution, telemetry tags и documented presets делают библиотеку предсказуемым инструментом в реальном проекте, а не локальным proof-of-concept.

**Это API превращает LangGraph из библиотеки "цепочек" в полноценный фреймворк для создания "умных систем управления", соответствующих стандартам кибернетики.**
