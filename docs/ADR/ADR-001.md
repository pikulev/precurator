# ADR 001: Архитектура автономной итеративной системы управления «Интеллектуальный Прогноз-Корректор» на базе LangGraph

**Статус:** Approved with refinements  
**Версия:** 1.1  
**Контекст:** Реализация системы адаптивного управления сложными объектами/процессами в условиях неопределенности с использованием LLM в качестве когнитивного регулятора.

---

## 1. Постановка задачи (The Problem)
Необходима система, способная автономно переводить объект из начального состояния $S_0$ в целевое состояние $S_{target}$ через последовательность управляющих воздействий $U$. 
**Проблема:** Линейные алгоритмы не справляются с динамическими изменениями и галлюцинациями LLM. 
**Решение:** Построение замкнутого контура управления с обратной связью по «Вектору ошибки».

---

## 2. Архитектурные принципы (Design Principles)

1.  **Полная Функция Управления (ПФУ):** Система должна самостоятельно распознавать факторы среды, формировать вектор целей и концепцию управления.
2.  **Схема «Прогноз-Корректор»:** Управляющее воздействие выбирается не только на основе текущей ошибки, но и на основе прогноза будущего состояния.
3.  **Минимизация Вектора Ошибки:** Цель работы графа — сведение нормы разности между целевым и текущим состоянием к значению меньше заданного порога $\epsilon$.
4.  **Устойчивость по Ляпунову:** Система должна иметь механизмы демпфирования (сдерживания), чтобы избежать автоколебаний и «разноса» (бесконечных циклов).
5.  **Иммутабельность по умолчанию:** Каждый шаг должен формировать новый снимок состояния, пригодный для отката, ветвления и воспроизводимого анализа.

---

## 3. Модель состояния (Graph State)

В LangGraph `State` выступает как **Пространство состояний** системы:

```python
class SystemState(TypedDict):
    # --- Вектор Целей (ДОТУ) ---
    target_vector: Dict[str, Any]      # Жесткие и мягкие параметры цели

    # --- Вектор Состояния (ТАУ) ---
    current_state: Dict[str, Any]      # Текущие метрики объекта
    world_context: Dict[str, Any]      # Модель мира / база знаний / ограничения среды

    # --- Вектор Ошибки ---
    error_vector: Dict[str, Any]       # Машиночитаемое рассогласование по метрикам
    error_norm: float                  # Количественная оценка ошибки (0.0 - 1.0)

    # --- Память, аудит и прогноз ---
    short_term_memory: Annotated[list, add]  # Последние N шагов для prompt-контекста
    audit_log_pointer: str | None      # Ссылка на полный журнал вне prompt
    prediction: str                    # Ожидаемый результат следующего шага
    stop_reason: str | None            # Причина завершения цикла

    # --- Управление выполнением ---
    iteration_count: int               # Счетчик итераций (дискретное время k)
    max_iterations: int                # Лимит (Запас устойчивости)
    convergence_rate: List[float]      # Динамика изменения ошибки (Delta Error)
    thread_id: str                     # Идентификатор потока / чекпоинтов LangGraph
    best_checkpoint_id: str | None     # Лучший известный снимок для time-travel
    metadata: Dict[str, Any]           # Trace IDs, user IDs, системный контекст
```

`history` в исходной версии признан слишком "тяжелым" и должен быть расщеплен на:

*   `short_term_memory` для локального контекста регулятора.
*   внешний `audit_log`, доступный через `audit_log_pointer`.

---

## 4. Структура графа (Nodes & Logic)

### 4.1. Узел «Observer» (Сенсор/Детектор)
*   **Функция:** Сбор данных из внешней среды или инструментов.
*   **Задача в ТАУ:** Формирование вектора обратной связи.
*   **Output:** Обновленный `current_state`.

### 4.2. Узел «Comparator» (Анализатор ПФУ)
*   **Функция:** Вычисление «Вектора ошибки».
*   **Логика:** Предпочтительно детерминированный компаратор; LLM используется только там, где сравнение нельзя выразить формальной метрикой.
*   **Критерий:** Если `error_norm < epsilon` — переход в `END`.
*   **Демпфирование:** Если ошибка не уменьшается 3 итерации подряд, узел сигнализирует о «застревании» (управленческий тупик).
*   **Требование:** `error_vector` должен оставаться структурированным и пригодным для построения графиков сходимости без парсинга текста.

### 4.3. Узел «Predictor-Strategist» (Регулятор)
*   **Функция:** Выработка концепции управления.
*   **Принцип:** Model Predictive Control (MPC). LLM анализирует `short_term_memory`, `world_context` и `error_vector`, формулирует действие $U(k)$ и **обязательно** дает прогноз: «После этого действия параметр X должен уменьшиться на Y».
*   **Output:** `action`, `prediction`.

### 4.4. Узел «Effector» (Исполнитель)
*   **Функция:** Реализация управляющего воздействия.
*   **Инструменты:** Tool Calling (API, БД, Код, Текст).
*   **Output:** Реальный результат действия.

### 4.5. Узел «Validator / External Verifier» (Внешний наблюдатель)
*   **Функция:** Проверка того, что прогноз регулятора и фактический результат действительно согласованы.
*   **Принцип:** Не должен полагаться на тот же источник оценки, что и `Predictor`; допускается отдельная модель, правило или deterministic check.
*   **Output:** `prediction_error`, сигнал об осцилляции, недостоверности или ложном улучшении.

---

## 5. Алгоритм управления (Control Flow)

1.  **Вход:** Инициализация `target_vector`.
2.  **Цикл:**
    *   **Step A:** `Observer` считывает реальность.
    *   **Step B:** `Comparator` определяет разрыв. Если разрыв приемлем — **STOP**.
    *   **Step C:** `Predictor` выбирает оптимальное воздействие, минимизирующее риск «перелета» (overshoot).
    *   **Step D:** `Effector` применяет изменения.
    *   **Step E:** `Validator` сравнивает «Прогноз» Предиктора с «Реальностью» Эффектора. Разница заносится в `short_term_memory`, а полный след фиксируется во внешнем `audit_log`, доступном через `audit_log_pointer`.
    *   **Step F:** Система сохраняет новый checkpoint и при необходимости обновляет `best_checkpoint_id`.
3.  **Выход:** Достижение цели, остановка по `StopPolicy` или остановка по `max_iterations` (аварийный режим).

---

## 6. Механизмы устойчивости (Stability & Robustness)

1.  **Зона нечувствительности (Dead Zone):** Если ошибка менее 2-3%, мы прекращаем итерации, чтобы не тратить ресурсы на «шум».
2.  **Анти-винд-ап (Anti-windup):** Если журнал воздействий слишком длинный, он начинает «давить» на LLM. Система должна суммаризировать только `short_term_memory`, оставляя полный `audit_log`, доступный через `audit_log_pointer`, вне prompt.
3.  **Коэффициент усиления (Gain Control):** Промпт Предиктора содержит инструкцию: *"Если ошибка велика — действуй радикально. Если ошибка мала — действуй ювелирно"*.
4.  **Проверка на осцилляции:** Если `error_norm` скачет (0.1 -> 0.8 -> 0.1), `Comparator` и `Validator` принудительно снижают «темп» управления, требуя от Предиктора более консервативных стратегий.
5.  **Политика остановки (StopPolicy):** Остановка определяется не только через $\epsilon$, но и через стоимость, лимит итераций, лимит токенов, отсутствие прогресса и внешние бизнес-ограничения.
6.  **Изоляция симуляций:** Воображаемые итерации выполняются в ветвящемся состоянии и не загрязняют основной контур управления.

---

## 7. Последствия (Consequences)

*   **Плюсы:** Высочайшая автономность, устойчивость к неверным ответам LLM, прозрачность процесса управления, возможность time-travel и отката к лучшему состоянию.
*   **Минусы:** Повышенный расход токенов, задержка (latency) из-за циклической природы, необходимость проектировать структурированные метрики ошибки и внешний контур верификации.

---

## 8. Метрики успеха (KPI)
1.  **Время сходимости:** Количество итераций до достижения цели.
2.  **Точность управления:** Остаточная величина вектора ошибки.
3.  **Устойчивость:** Отсутствие циклов, где ошибка бесконечно растет.
4.  **Цена управления:** Стоимость токенов, инструментальных вызовов и времени до остановки по `StopPolicy`.
