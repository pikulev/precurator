# ADR 001: Архитектура автономной итеративной системы управления «Интеллектуальный Прогноз-Корректор» на базе LangGraph

**Статус:** Approved with refinements  
**Версия:** 1.1  
**Контекст:** Реализация системы адаптивного управления сложными объектами/процессами в условиях неопределенности с использованием LLM в качестве когнитивного регулятора.

---

## 1. Постановка задачи (The Problem)
Необходима система, способная автономно переводить объект из начального состояния $S_0$ в целевое состояние $S_{target}$ через последовательность управляющих воздействий $U$. 
**Проблема:** Линейные алгоритмы не справляются с динамическими изменениями и галлюцинациями LLM. 
**Решение:** Построение замкнутого контура управления с обратной связью по «Вектору ошибки».

---

## 2. Архитектурные принципы (Design Principles)

1.  **Полная Функция Управления (ПФУ):** Система должна самостоятельно распознавать факторы среды, формировать вектор целей и концепцию управления.
2.  **Схема «Прогноз-Корректор»:** Управляющее воздействие выбирается не только на основе текущей ошибки, но и на основе прогноза будущего состояния.
3.  **Минимизация Вектора Ошибки:** Цель работы графа — сведение нормы разности между целевым и текущим состоянием к значению меньше заданного порога $\epsilon$.
4.  **Устойчивость по Ляпунову:** Система должна иметь механизмы демпфирования (сдерживания), чтобы избежать автоколебаний и «разноса» (бесконечных циклов).
5.  **Иммутабельность по умолчанию:** Каждый шаг должен формировать новый снимок состояния, пригодный для отката, ветвления и воспроизводимого анализа.

---

## 3. Модель состояния (Graph State)

В LangGraph `State` выступает как **пространство состояний** системы, но после ревью оно должно быть **декомпозировано на два слоя**:

*   `control` — предметное состояние и вычислимые сигналы контура управления.
*   `runtime` — оркестрационные, checkpoint, telemetry и HITL-метаданные исполнения.

```python
class ControlBasis(TypedDict):
    # --- Вектор Целей (ДОТУ) ---
    target_vector: Dict[str, Any]      # Жесткие и мягкие параметры цели

    # --- Вектор Состояния (ТАУ) ---
    current_state: Dict[str, Any]      # Текущие метрики объекта
    world_context: Dict[str, Any]      # Модель мира / база знаний / ограничения среды

    # --- Вектор Ошибки ---
    error_vector: Dict[str, float | int | str | bool | None]
    error_norm: float                  # Количественная оценка ошибки (0.0 - 1.0)
    delta_error: float                 # Производная ошибки относительно k-1
    error_trend: str                   # improving | flat | degrading | oscillating
    convergence_rate: List[float]      # История delta_error для guards и графиков

    # --- Память и прогноз ---
    short_term_memory: List[Dict[str, Any]]  # Bounded window, а не бесконечный append-only log
    prediction: str                    # Ожидаемый результат следующего шага


class RuntimeState(TypedDict):
    k: int                             # Дискретное время (номер итерации)
    max_iterations: int                # Лимит (Запас устойчивости)
    status: str                        # optimizing | converged | stuck | failed | awaiting_human_intervention | aborted
    stop_reason: str | None            # Причина завершения цикла/паузы
    diagnostics: Dict[str, Any] | None # Структурированное объяснение stuck/failure/interrupt
    audit_log_pointer: str | None      # Ссылка на полный журнал вне prompt
    checkpoint_id: str | None          # Текущий checkpoint
    best_checkpoint_id: str | None     # Лучший известный снимок для time-travel
    simulation: bool                   # Признак ветки симуляции
    human_decision: Dict[str, Any] | None
    metadata: Dict[str, Any]           # Trace IDs, user IDs, системный контекст


class SystemState(TypedDict):
    control: ControlBasis
    runtime: RuntimeState
```

Правило после ревью: `runtime`-поля не являются источником истины о предметном состоянии объекта и не должны подменять собой `control.current_state`, `control.error_vector` или `control.target_vector`.

`history` в исходной версии признан слишком "тяжелым" и должен быть расщеплен на:

*   bounded `short_term_memory` для локального контекста регулятора.
*   внешний `audit_log`, доступный через `runtime.audit_log_pointer`.

Краткосрочная память не может расти бесконечно. Когда окно переполняется, система обязана выполнить детерминированный `Summarizer / Compressor`, который сворачивает старые шаги в компактное резюме, не разрушая полный audit trail.

После ревью это поведение должно рассматриваться не как "внутренняя магия" графа, а как явный контракт: `CompactionStrategy(history: ControlStep[]) -> ShortTermMemoryWindow`. Стратегия обязана быть детерминированной, не иметь side effects, сохранять ссылочную целостность с внешним `audit_log` и явно определять, какие шаги заменяются summary, а какие остаются в рабочем окне.

Дополнительные инварианты publish-ready версии:

1.  **Сериализуемость state:** `SystemState`, включая `control` и `runtime`, должен быть совместим с JSON/checkpoint serialization. Инстансы моделей, closures, клиенты SDK, классы инструментов и иные несериализуемые объекты не могут попадать в graph state.
2.  **Жесткий контракт базиса:** Пользовательский LangGraph-узел обязан получать предсказуемую структуру `state.control` и `state.runtime`, а библиотека не должна прятать критические поля управления в неявных envelope-объектах.
3.  **Разрешение ссылок вне state:** Функции, модели и tool handlers подключаются через внешний runtime registry или реестры ссылок, а не через встраивание executable-объектов в checkpointed state.
4.  **Проверяемость до исполнения:** Любое состояние, не соответствующее контракту `ControlBasis`, должно быть отклонено до входа в дорогостоящий reasoning-цикл и до отправки токенов в LLM.

## 4. Структура графа (Nodes & Logic)

### 4.1. Узел «Evolve» (Шаг эволюции / Сенсор-исполнитель)
*   **Функция:** Сбор данных из внешней среды или инструментов, а при необходимости и продвижение системы к следующему `current_state`.
*   **Задача в ТАУ:** Формирование вектора обратной связи.
*   **Output:** Обновленный `current_state`.

### 4.2. Узел «Comparator» (Анализатор ПФУ)
*   **Функция:** Вычисление «Вектора ошибки».
*   **Логика:** Предпочтительно детерминированный компаратор; LLM используется только там, где сравнение нельзя выразить формальной метрикой.
*   **Output:** `error_vector`, `error_norm`, `delta_error`, `error_trend`.
*   **Критерий:** Если `error_norm < epsilon` — переход в `END`.
*   **Демпфирование:** Если ошибка не уменьшается 3 итерации подряд, `delta_error > 0` или `error_trend = oscillating`, узел сигнализирует о «застревании» либо переводит систему в более консервативный режим.
*   **Требование:** `error_vector` должен оставаться структурированным и пригодным для построения графиков сходимости без парсинга текста.
*   **Диагностика:** При переходе в `stuck`, `failed` или `awaiting_human_intervention` компаратор и/или валидатор должны уметь заполнять `runtime.diagnostics` структурированным объяснением причины: код, сообщение, наблюдаемые признаки, повторяющееся действие, окно осцилляции, рекомендуемое вмешательство.

### 4.3. Узел «Predictor-Strategist» (Регулятор)
*   **Функция:** Выработка концепции управления.
*   **Принцип:** Model Predictive Control (MPC). LLM анализирует `short_term_memory`, `world_context` и `error_vector`, формулирует действие $U(k)$ и **обязательно** дает прогноз: «После этого действия параметр X должен уменьшиться на Y».
*   **Output:** `action`, `prediction`.

### 4.4. Узел «Effector» (Исполнитель)
*   **Функция:** Реализация управляющего воздействия.
*   **Инструменты:** Tool Calling (API, БД, Код, Текст).
*   **Output:** Реальный результат действия.

### 4.5. Узел «Validator / External Verifier» (Внешний наблюдатель)
*   **Функция:** Проверка того, что прогноз регулятора и фактический результат действительно согласованы.
*   **Принцип:** Не должен полагаться на тот же источник оценки, что и `Predictor`; допускается отдельная модель, правило или deterministic check.
*   **Output:** `prediction_error`, `delta_error`, сигнал об осцилляции, недостоверности или ложном улучшении.
*   **HITL-эскалация:** При критическом рассогласовании, policy violation или high-risk action валидатор должен иметь право инициировать `interrupt` и перевести систему в `runtime.status = 'awaiting_human_intervention'`.

## 5. Алгоритм управления (Control Flow)

1.  **Вход:** Инициализация `control.target_vector`.
2.  **Цикл:**
    *   **Step A:** `Evolve` считывает реальность и при необходимости выполняет effectful шаг, после чего возвращает следующий `current_state`.
    *   **Step B:** `Comparator` определяет разрыв и его динамику (`error_norm`, `delta_error`, `error_trend`). Если разрыв приемлем — **STOP**.
    *   **Step C:** `Predictor` выбирает оптимальное воздействие, минимизирующее риск «перелета» (overshoot).
    *   **Step D:** `Effector` применяет изменения.
    *   **Step E:** `Validator` сравнивает «Прогноз» Предиктора с «Реальностью» Эффектора. Разница заносится в bounded `short_term_memory`, а полный след фиксируется во внешнем `audit_log`, доступном через `runtime.audit_log_pointer`.
    *   **Step F:** Если `Verifier` или `StopPolicy` обнаруживают критический риск, система выполняет `interrupt`, сохраняет checkpoint и переходит в `awaiting_human_intervention` до получения `resume`, `abort` или `human_decision`.
    *   **Step G:** Система сохраняет новый checkpoint и при необходимости обновляет `runtime.best_checkpoint_id`.
3.  **Выход:** Достижение цели, остановка по `StopPolicy`, остановка по `max_iterations` (аварийный режим) или пауза для Human-in-the-loop.

---

## 6. Механизмы устойчивости (Stability & Robustness)

1.  **Зона нечувствительности (Dead Zone):** Если ошибка менее 2-3%, мы прекращаем итерации, чтобы не тратить ресурсы на «шум».
2.  **Анти-винд-ап (Anti-windup):** Если журнал воздействий слишком длинный, он начинает «давить» на LLM. Система должна суммаризировать только bounded `short_term_memory`, оставляя полный `audit_log`, доступный через `runtime.audit_log_pointer`, вне prompt.
3.  **Коэффициент усиления (Gain Control):** Промпт Предиктора содержит инструкцию: *"Если ошибка велика — действуй радикально. Если ошибка мала — действуй ювелирно"*.
4.  **Проверка на осцилляции:** Если `error_norm` скачет (0.1 -> 0.8 -> 0.1), `delta_error` остается положительной или `error_trend = oscillating`, `Comparator` и `Validator` принудительно снижают «темп» управления, требуя от Предиктора более консервативных стратегий.
5.  **Политика остановки (StopPolicy):** Остановка определяется не только через $\epsilon$, но и через стоимость, лимит итераций, лимит токенов, отсутствие прогресса, внешние бизнес-ограничения и внешнюю команду `abort`.
6.  **Human-in-the-loop:** Для критических сценариев должна существовать внешняя кнопка `STOP/ABORT`, а также ветка `interrupt -> awaiting_human_intervention -> resume|abort`.
7.  **Изоляция симуляций:** Воображаемые итерации выполняются в ветвящемся состоянии и не загрязняют основной контур управления. В simulation-ветке запрещены реальные side effects; допустимы только mock/dry-run инструменты. Все журналы, checkpoints и telemetry такой ветки обязаны явно маркироваться как `simulation`.

Для интеграции с LangGraph пауза и возобновление должны опираться на штатный `checkpointer`, а не на самодельное хранение "подвешенного" состояния вне графа. Библиотека может предоставлять ergonomic API для `approve`, `override` и `resume`, но источником истины для interrupt/resume остается checkpointed state.

Нормативные требования интеграционной готовности:

1.  `interrupt_before` / `interrupt_after` должны сохранять инвариант дискретного времени: `runtime.k` не теряется и не "перескакивает" при паузе, ручном решении и последующем `resume`.
2.  Совместимость с checkpointers должна предполагать отсутствие в checkpointed state скрытых class instances, промисов, открытых дескрипторов и иных объектов, которые невозможно надежно восстановить через `SqliteSaver`, `PostgresSaver` или эквивалентный backend.
3.  `best_checkpoint_id`, `checkpoint_id`, `stop_reason` и `human_decision` должны трактоваться как часть runtime-аудита и восстановления, а не как замена предметного `control.current_state`.
4.  Любая publish-ready реализация обязана детерминированно останавливать цикл по `epsilon`, `max_iterations`, `max_token_budget`, policy violation или внешнему `abort`, не полагаясь на "самозатухание" промпта как единственный guard.

---

## 7. Последствия (Consequences)

*   **Плюсы:** Высочайшая автономность, устойчивость к неверным ответам LLM, прозрачность процесса управления, возможность time-travel и отката к лучшему состоянию.
*   **Минусы:** Повышенный расход токенов, задержка (latency) из-за циклической природы, необходимость проектировать структурированные метрики ошибки и внешний контур верификации.

---

## 8. Метрики успеха (KPI)
1.  **Время сходимости:** Количество итераций до достижения цели.
2.  **Точность управления:** Остаточная величина вектора ошибки.
3.  **Устойчивость:** Отсутствие циклов, где ошибка бесконечно растет.
4.  **Цена управления:** Стоимость токенов, инструментальных вызовов и времени до остановки по `StopPolicy`.
