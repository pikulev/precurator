# ADR 003: Memory Management & Compaction для итеративного control loop

**Статус:** Approved with refinements  
**Автор:** System Architect / Control Theory Expert  
**Контекст:** Итеративные LangGraph-системы накапливают историю шагов и быстро упираются в лимиты токенов, деградацию внимания LLM и рост стоимости исполнения.

---

## 1. Проблема

`precurator` проектируется как библиотека для замкнутого цикла "наблюдение -> сравнение -> прогноз -> действие -> валидация". В такой системе история важна не сама по себе, а как сигнал о динамике ошибки, повторяющихся неудачах и характере сходимости.

Если библиотека хранит историю как бесконечный append-only log внутри prompt-facing state, возникают четыре системные проблемы:

1.  **Context stuffing:** размер prompt растет быстрее полезного сигнала.
2.  **Drift of attention:** LLM начинает учитывать устаревшие или шумные эпизоды сильнее, чем актуальное окно управления.
3.  **Cost explosion:** стоимость шага растет непропорционально числу итераций.
4.  **Operational fragility:** checkpointed state становится тяжелым, а поведение на длинных сериях теряет предсказуемость.

Следовательно, bounded memory для control loop не является optional optimization. Это обязательный архитектурный контракт publish-ready библиотеки.

---

## 2. Решение

Библиотека разделяет память на два слоя:

1.  **`shortTermMemory.steps`:** небольшое рабочее окно последних шагов в подробном виде.
2.  **`shortTermMemory.summary`:** компактное представление более ранней истории, уже вытесненной из окна.

Полный audit trail не должен храниться в prompt-facing state. Он выносится во внешний журнал, связанный через `runtime.auditLogRef` или эквивалентный указатель.

Память компактизируется не по "магии" модели, а через явный контракт компакции, который должен быть детерминированным, side-effect free и совместимым с checkpoint serialization.

---

## 3. Цели и не-цели

### 3.1. Цели

1.  Сохранить для `Predictor` и `Verifier` только релевантный сигнал о недавней динамике.
2.  Ограничить рост prompt и токен-бюджета на длинных итеративных сериях.
3.  Снизить риск осцилляций, вызванных "забыванием" прошлых неудачных стратегий.
4.  Сохранить совместимость с `LangGraph` checkpoints и `interrupt/resume`.

### 3.2. Не-цели

1.  Не пытаться хранить полный raw transcript инструментов внутри `shortTermMemory`.
2.  Не подменять compaction внешним observability/audit storage.
3.  Не делать memory strategy скрытой эвристикой, недоступной из конфигурации системы.

---

## 4. Контракт состояния памяти

ADR-003 уточняет поведение `ShortTermMemoryWindow`, введенного в ADR-002.

```typescript
type CompactionStrategy = 'sliding-window' | 'summarize-oldest' | 'hybrid';

interface ShortTermMemoryWindow {
  steps: ControlStep[];
  maxShortTermSteps: number;
  compactionStrategy: CompactionStrategy;
  summaryReplacementSemantics: 'replace-compacted-steps' | 'replace-all-but-recent-window';
  summary?: string;
}

interface CompactionContext {
  maxShortTermSteps: number;
  strategy: CompactionStrategy;
  semantics: 'replace-compacted-steps' | 'replace-all-but-recent-window';
  auditLogRef?: string;
}

type CompactShortTermMemory = (
  history: ControlStep[],
  context: CompactionContext
) => ShortTermMemoryWindow;
```

Нормативные требования к этому контракту:

1.  `steps` всегда содержит только актуальное bounded window.
2.  `summary` описывает только уже вытесненные шаги и не дублирует текущее окно без причины.
3.  `CompactShortTermMemory` не может менять внешний мир, вызывать инструменты или зависеть от скрытого mutable state.
4.  Результат компакции должен быть сериализуемым и пригодным для сохранения в checkpoint.

---

## 5. Стратегии компакции

Библиотека должна поддерживать как минимум три стратегии.

### 5.1. `sliding-window`

Хранить только последние `K` шагов, отбрасывая более старые.

Плюсы:
1.  Максимальный детерминизм.
2.  Минимальная стоимость.
3.  Простая совместимость с checkpoints.

Минусы:
1.  Система может забыть ранние провальные траектории.
2.  Растет риск повторных циклов и локальной осцилляции.

### 5.2. `summarize-oldest`

При переполнении окна старые шаги сворачиваются в `summary`, а последние `K` итераций остаются детализированными.

Плюсы:
1.  Сохраняется семантика прошлых ошибок и неудачных действий.
2.  `Predictor` лучше видит анти-паттерны и причины деградации.

Минусы:
1.  Нужна дополнительная операция свертки.
2.  Плохо спроектированное summary может искажать историю.

### 5.3. `hybrid`

Комбинировать небольшое детализированное окно с выборочным сохранением ключевых шагов или семантической сверткой.

Плюсы:
1.  Баланс между стоимостью и памятью о важных событиях.
2.  Лучше подходит для длинных траекторий с редкими, но значимыми скачками `deltaError`.

Минусы:
1.  Сложнее для объяснения и тестирования.
2.  Требует четкой политики отбора ключевых шагов.

---

## 6. Триггер и место в control flow

Компакция должна рассматриваться как штатный этап жизненного цикла состояния.

Рекомендуемый порядок:

1.  `Evolve` обновляет наблюдаемое состояние и при необходимости завершает effectful шаг.
2.  `Comparator` и `Verifier` вычисляют сигналы ошибки и подтверждают прогресс.
3.  `Effector` завершает действие или dry-run ветку.
4.  Новый `ControlStep` добавляется в `shortTermMemory.steps`.
5.  Если размер окна превышает `maxShortTermSteps`, выполняется `CompactShortTermMemory`.
6.  Обновленный `ShortTermMemoryWindow` попадает в checkpointed state.

Важно: compaction происходит **до** следующего шага reasoning, чтобы `Predictor` видел уже ограниченное и нормализованное memory window.

---

## 7. Инварианты интеграции с LangGraph

1.  Память, попадающая в graph state, должна оставаться совместимой с `checkpointer` и корректно восстанавливаться после `interrupt`.
2.  `interrupt_before` / `interrupt_after` не должны разрушать связь между `runtime.k`, `shortTermMemory.steps` и `summary`.
3.  В simulation branch compaction может выполняться, но ее результат обязан оставаться изолированным от основного `best_checkpoint_id` и "боевого" audit trail.
4.  Компакция не должна зависеть от наличия live model instance внутри state; если для summary используется LLM, она вызывается через runtime registry, а в state сохраняется только результат.

---

## 8. Safety-инварианты для бизнеса

Publish-ready реализация memory management обязана поддерживать следующие гарантии:

1.  **Bounded growth:** prompt-facing memory не растет бесконечно.
2.  **Deterministic stop-loss compatibility:** `maxIterations`, `maxTokenBudget` и `epsilon` продолжают работать предсказуемо и не размываются "тяжелой" памятью.
3.  **Audit separation:** полная трасса событий сохраняется вне prompt, а summary не считается единственным источником истины.
4.  **Anti-oscillation support:** память должна сохранять достаточно сигнала, чтобы `Verifier` и guards могли различать improving, degrading и oscillating trajectories.
5.  **Testability:** логика компакции должна тестироваться на синтетических последовательностях без обязательного вызова реальной LLM.

---

## 9. Последствия

Плюсы:
1.  Устойчивость на длинных сериях итераций.
2.  Снижение стоимости reasoning на поздних шагах.
3.  Более предсказуемое поведение при checkpoint/resume.
4.  Явный архитектурный контракт вместо скрытой "магии памяти".

Минусы:
1.  Появляется отдельный слой проектирования и тестирования memory strategy.
2.  Для `summarize-oldest` и `hybrid` нужно контролировать качество summary и дрейф смысла.

---

## 10. Нормативный вывод

`precurator` не считается publish-ready без явно определенной стратегии bounded memory. Для публичного npm-пакета memory compaction является не optional feature, а частью базового operational contract: система должна сохранять рассудок, цену и воспроизводимость на длинных траекториях управления.
