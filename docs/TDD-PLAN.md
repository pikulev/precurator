# Стандарт тестирования и план TDD для `precurator`

## 1. Пререквизиты (Environment & Tooling)

Для запуска цикла Red-Green-Refactor необходимо настроить следующую инфраструктуру:

### 1.1. Стек инструментов
*   **Vitest:** Основной фреймворк для тестов (поддержка ESM и высокая скорость).
*   **tsd / expect-type:** Для тестирования контрактов TypeScript (Generics `TTarget`, `TCurrent`).
*   **Zod:** Для валидации схем и очистки (stripping) данных.
*   **LangGraph `MemorySaver`:** Для интеграционных тестов `interrupt/resume`.

### 1.2. Тестовые хелперы (Custom Matchers)
Перед началом разработки должны быть реализованы два системных хелпера:
1.  **`toBeSerializable()`:** Проверяет, что объект является POJO (Plain Old JS Object), не содержит функций, промисов или скрытых классов.
2.  **`assertStrictSchema(schema, data)`:** Проверяет, что данные не содержат лишних полей (защита от галлюцинаций LLM через `.strip()` или `.strict()`).

### 1.3. Mocking Strategy
*   **Deterministic Predictor:** Заглушка, которая вместо LLM меняет `current_state` по математической формуле (например, `current + (target - current) * 0.5`). Используется для тестов сходимости графа.
*   **Registry Mock:** Объект, имитирующий внешние зависимости (модели, инструменты) для проверки биндинга конфига.

## 2. Лучшие практики разработки (Guardrails)

1.  **Принцип "Чистого редьюсера":** Тестируйте обновление состояния (`State Reducers`) как чистые синхронные функции отдельно от асинхронных вызовов узлов графа.
2.  **Изоляция конфигурации:** Тест считается успешным только если `ControlSystemConfig` можно превратить в JSON и восстановить обратно без потери логики (кроме ссылок на функции).
3.  **Защита Simulation Branch:** Любой тест инструментов обязан проверять, что при `runtime.simulation: true` вызов деструктивных методов блокируется на уровне инфраструктуры.
4.  **Никаких "Token Counting" в Unit-тестах:** Вместо подсчета символов тестируйте **структуру данных**. Если размер массива `steps` сократился до лимита `K` — контракт памяти выполнен.

## 3. Поэтапный план TDD

### Слой 1: Базис состояния и Типизация (ADR-002)
*   **Test 1.1 (Type Logic):** Передача несовместимых `TTarget` и `TCurrent` в компилятор должна вызывать ошибку TS (через `tsd`).
*   **Test 1.2 (Serialization):** Попытка записи класса в `State` должна проваливать проверку `toBeSerializable`.
*   **Test 1.3 (Schema Stripping):** Подача объекта с "галлюцинаторными" полями в `Observer`. Проверка, что после валидации в стейте остаются только поля из Zod-схемы.

### Слой 2: Кибернетическая математика (ADR-001)
*   **Test 2.1 (Comparator):** Расчет `errorScore` [0, 1] для структурных данных.
*   **Test 2.2 (Trend):** Проверка смены статуса `errorTrend` при последовательности ошибок: `0.5 -> 0.3 -> 0.4` (должно быть `degrading`).
*   **Test 2.3 (Oscillation):** Детекция "дребезга" при скачках ошибки.

### Слой 3: Управление памятью (ADR-003)
*   **Test 3.1 (Sliding Window):** При `maxShortTermSteps: 5` и добавлении 6-го шага, 1-й должен исчезнуть.
*   **Test 3.2 (Summarization Contract):** Проверка, что функция компакции корректно вызывает внешний `Summarizer` через реестр и сохраняет результат в поле `summary`.
*   **Test 3.3 (Audit Continuity):** Проверка, что при удалении шагов из памяти `auditLogRef` (указатель на полный лог) остается неизменным.

### Слой 4: Оркестрация графа и Роутинг
*   **Test 4.1 (Routing Logic):** Функция `shouldContinue` должна возвращать `END`, если `errorScore < epsilon` или `status === 'failed'`.
*   **Test 4.2 (Stop Policies):** Принудительная остановка по `maxIterations` и `maxTokenBudget`, включая перенос накопленного бюджета через `resume`.
*   **Test 4.3 (Simulation Isolation):** Попытка вызова "опасного" инструмента в ветке симуляции должна выбрасывать исключение `SimulationSecurityError`.

### Слой 5: API, Реестр и HITL
*   **Test 5.1 (Registry Binding):** Создание системы из чистого JSON-конфига и `RuntimeRegistry`. Проверка, что ссылки (`modelRef`) превратились в рабочие инстансы.
*   **Test 5.2 (Interrupt/Resume):** Запуск графа, остановка на `interrupt`, ручной ввод `humanDecision`, возобновление. Проверка сохранения `runtime.k`.

## 4. Дорожная карта реализации

### Phase Alpha: "Foundation" (1-2 неделя)
*   Реализация Слоя 1 и 2.
*   **Результат:** Библиотека умеет валидировать входные данные и считать математику ошибки без графа.

### Phase Beta: "The Engine" (3-4 неделя)
*   Реализация Слоя 3 и 4.
*   **Результат:** Рабочий цикл LangGraph с ограниченной памятью и детерминированным Предиктором.

### Phase Gamma: "The Librarian" (5 неделя)
*   Реализация Слоя 5 и 6 (Телеметрия).
*   **Результат:** Поддержка HITL, интеграция с LangSmith, сериализуемые конфиги.

### Phase RC: "Stability" (6 неделя)
*   Нагрузочные тесты (профилирование памяти на 100+ итерациях).
*   Проверка всех инвариантов ADR на реальных моделях (GPT-4o, Claude 3.5).

---

**Нормативный вывод:** Библиотека считается готовой к релизу (Publish-Ready), если 100% тестов из данного плана проходят в CI, а `tsc` подтверждает отсутствие `any` в публичных контрактах.
